# Research: PDP Product Data Extraction Strategy

**Feature**: `004-pdp-product-data-strategy`  
**Date**: 2026-04-01

## R-001: ld+json Parsing Strategy

**Decision**: Query all `<script type="application/ld+json">` elements via `document.querySelectorAll`, parse each with `JSON.parse` inside try/catch, and identify the Product object by checking `@type`.

**Rationale**: DOM querying is synchronous, fast, and universally supported. VTEX stores may have multiple ld+json tags (BreadcrumbList, Organization, Product) — scanning all and filtering by type is the reliable approach. JSON.parse failure on any tag does not affect others.

**Alternatives considered**:
- **Single querySelector with content check**: Rejected — no CSS selector can filter by JSON content inside script tags.
- **Regex extraction**: Rejected — fragile, cannot handle nested JSON reliably.
- **MutationObserver for dynamic ld+json**: Rejected — ld+json tags are server-rendered and present at DOMContentLoaded; no need for dynamic observation.

**Schema.org type detection**:
- Check `data["@type"]` for `"Product"` or `"ProductGroup"` (string match).
- If `@type` is an array, check if it includes either type.
- If the root object has `@graph` (array), iterate entries and find the Product.
- If the root object has `mainEntity`, check if it's the Product.
- VTEX IO `structured-data` app typically outputs a flat `Product` or `ProductGroup` object.

**Field mapping from ld+json to ProductData**:

| ld+json field | ProductData field | Notes |
|---------------|-------------------|-------|
| `name` | `productName` | Direct string |
| `description` | `description` | Direct string |
| `brand.name` or `brand` (string) | `brand` | Handle both object and string |
| `additionalProperty[]` | `attributes` | Map `{name, value}` pairs; filter `valueReference === "SPECIFICATION"` |
| — | `linkText` | From URL slug (not in ld+json) |
| — | `account` | From account resolver |

**SKU context from ld+json**:
- `offers.offers[]` contains per-SKU pricing (`price`, `priceCurrency`, `availability`).
- `hasVariant[]` on `ProductGroup` contains variant entries with names and offers.
- Map to the same context string format as `buildProductContextString`.

## R-002: `__NEXT_DATA__` Extraction Strategy

**Decision**: Read `window.__NEXT_DATA__.props.pageProps.data.product` using optional chaining. This path was confirmed from a real VTEX FastStore product page JSON sample (Surface Go 3 on `demoanalyst.demoaccount19.com`).

**Rationale**: FastStore is built on Next.js, which serializes `getStaticProps`/`getServerSideProps` return values into `window.__NEXT_DATA__`. The product data is deeply nested but follows a consistent structure across FastStore stores because it's generated by the FastStore API GraphQL layer.

**Alternatives considered**:
- **Multiple path patterns**: Could also check `props.pageProps.product` (without `.data`). Decision: start with the confirmed path, add alternative paths if real-world stores show variations. Use a simple list of candidate paths tried in order.
- **Page type validation**: `__NEXT_DATA__.page === "/[slug]/p"` could confirm PDP. Decision: useful as a pre-check before deep traversal — adopt it as a guard.

**Field mapping from `__NEXT_DATA__` to ProductData**:

| `__NEXT_DATA__` field | ProductData field | Notes |
|-----------------------|-------------------|-------|
| `isVariantOf.name` (fallback: `name`) | `productName` | Product-level name preferred for caching |
| `description` | `description` | Direct string |
| `brand.name` | `brand` | Nested object |
| `customData.specificationGroups` | `attributes` | Filter internal properties (`sellerId`, etc.) using existing `INTERNAL_PROPERTIES` set |
| — | `linkText` | From URL slug or `__NEXT_DATA__.query.slug` |
| — | `account` | From account resolver |

**SKU context from `__NEXT_DATA__`**:
- `offers.offers[]` contains pricing for the current SKU.
- `isVariantOf.skuVariants.allVariantProducts[]` lists all variants with names and IDs.
- `isVariantOf.skuVariants.availableVariations` maps variation names to values.
- Map to the same context string format, though with less per-SKU pricing detail than the IS API (only the current SKU's offer is fully detailed).

**`__NEXT_DATA__` staleness on SPA navigation**: On FastStore SPA navigations, Next.js updates `__NEXT_DATA__` via client-side routing. The waterfall re-runs on each navigation event, so it reads the updated global each time.

## R-003: Waterfall Orchestration Pattern

**Decision**: Implement as a single async function `resolveProductData(slug, account)` that tries each strategy sequentially and returns the first valid result plus the raw product object (for context string building).

**Rationale**: Sequential waterfall is simpler than parallel-with-cancellation. The first two strategies are synchronous (< 1ms), so the waterfall adds negligible overhead before reaching the async IS API fallback.

**Alternatives considered**:
- **Strategy pattern with registry**: Rejected — over-engineered for three fixed strategies. A simple sequential function is more readable.
- **Parallel execution with first-to-resolve**: Rejected — would always trigger the IS API network request even when unnecessary, defeating the purpose of avoiding API calls.
- **Plugin/middleware chain**: Rejected — adds abstraction without benefit for a fixed three-step chain.

**Validation gate**: Each strategy's result is validated by `isValidProductData(data)` — checks `productName` is truthy AND at least one of `description` or `brand` is non-empty.

**Return shape**: `{ productData, rawProduct, source }` where:
- `productData` is the normalized `ProductData` payload for starters.
- `rawProduct` is the source-specific raw product object for building the context string.
- `source` is `'ld+json' | 'next-data' | 'intelligent-search'` for debugging.

## R-004: Account Name Resolution

**Decision**: `window.__RUNTIME__?.account || window.VTEX_METADATA?.account`. No further fallback.

**Rationale**:
- `window.__RUNTIME__` is the standard runtime object on VTEX IO stores, always populated server-side.
- `window.VTEX_METADATA` is documented for FastStore stores: `{ account: storeConfig.api.storeId, renderer: 'faststore' }`.
- The hostname fallback (`window.location.hostname.split('.')[0]`) was unreliable for custom domains (e.g., `shop.mybrand.com` would yield `shop`, not the VTEX account name).
- If neither object exists, the function returns `undefined`, and the waterfall skips starters generation since `account` is required for the payload.

**Alternatives considered**:
- **Keep hostname fallback as tertiary**: Rejected per spec requirement FR-010 — hostname is unreliable and the user explicitly requested its removal.
- **Extract from `__NEXT_DATA__` meta/canonical URL**: Rejected — the canonical URL contains the store domain, not the account name.

## R-005: Context String Building per Source

**Decision**: Adapt `buildProductContextString` to work with all three data shapes by creating source-specific adapters that normalize to a common intermediate format, then feed into the existing string builder.

**Rationale**: The context string format must remain identical regardless of source (FR-017). Rather than duplicating string building logic per source, normalize the raw product data into a common shape that `buildProductContextString` already understands.

**Alternatives considered**:
- **Three separate context builders**: Rejected — duplicates string formatting logic and risks format drift.
- **Modify `buildProductContextString` to accept any shape**: Rejected — makes the function complex with shape detection. Better to normalize before calling.

**Normalization approach**:
1. **IS API** → already in the expected format (no change needed).
2. **ld+json** → map `offers.offers[]` to `items[]` with `{ nameComplete, sellers: [{ commertialOffer: { Price, AvailableQuantity } }], variations: [] }`.
3. **`__NEXT_DATA__`** → map `isVariantOf.skuVariants.allVariantProducts[]` to `items[]`, enriching with offers data where available. Current SKU has full offer data; other variants have name and ID only.

## R-006: Testing Strategy

**Decision**: Pure unit tests for each extractor function. Integration-level tests for the waterfall. Mock DOM for ld+json (inject `<script>` tags into jsdom), mock `window.__NEXT_DATA__`, mock `fetch` for IS API.

**Test categories**:
1. **`extractFromLdJson(slug)`** — Unit tests: valid Product, valid ProductGroup, `@graph` wrapper, `mainEntity` wrapper, malformed JSON, missing `@type`, missing required fields, multiple tags, no product tag.
2. **`extractFromNextData(slug)`** — Unit tests: valid data at confirmed path, missing `__NEXT_DATA__`, missing nested product, `isVariantOf.name` present, fallback to `name`, invalid/incomplete data, `customData.specificationGroups` filtering.
3. **`resolveProductData(slug, account)`** — Integration tests: ld+json succeeds (no further calls), ld+json fails + `__NEXT_DATA__` succeeds, both fail + IS API succeeds, all fail returns null.
4. **`getVtexAccount()`** — Unit tests: `__RUNTIME__` present, `VTEX_METADATA` present, neither present.
5. **`isValidProductData(data)`** — Unit tests: all combinations of name/description/brand presence.
6. **Context string building** — Unit tests: verify equivalent output from each source shape.

**Coverage target**: 80%+ on all four metrics (statements, branches, functions, lines).
