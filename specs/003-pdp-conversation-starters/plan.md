# Implementation Plan: PDP Conversation Starters

**Branch**: `003-pdp-conversation-starters` | **Date**: 2026-03-14 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/003-pdp-conversation-starters/spec.md`

## Summary

Implement conversation starters for VTEX product pages (PDPs) in the webchat-react frontend. The feature automatically detects PDP pages, extracts product data via the VTEX Intelligent Search API, requests AI-generated questions through the existing `@weni/webchat-service` WebSocket bridge (`getStarters`/`clearStarters`), and displays up to 3 clickable buttons in two variants: floating above the launcher (compact) and inside the chat (full). Additionally, the widget sets SKU-level context for AI responses, monitors SPA navigation, and exposes a manual `setConversationStarters` public API. Mobile viewports auto-hide floating starters after 5 seconds.

## Design Decisions

### D1: Hook-Based Architecture

The `useConversationStarters` hook encapsulates all starters logic:
- Service event listeners (`starters:received`, `starters:error`)
- PDP detection on mount and navigation
- VTEX API calls for product data
- SKU context management
- Mobile auto-hide timer
- Starters state (`questions`, `source`, `isLoading`, `isCompactVisible`, `isDismissed`)

This follows the existing pattern of `useWeniChat` and voice mode hooks.

### D2: Compact Variant Placement

The compact variant is rendered in `WidgetContent` (inside `Widget.jsx`) between the `Chat` component and the `Launcher` component. Since `Widget` uses `flex-direction: column; align-items: flex-end; justify-content: flex-end`, a component placed before `Launcher` in the JSX naturally appears above the launcher, matching the design.

### D3: Full Variant Placement

The full variant is rendered at the bottom of `MessagesList` (after message groups, before the scroll anchor) with `margin-top: auto` when:
- Starters are available (`questions.length > 0`)
- Starters are not dismissed

This placement works for both empty chats (starters push to bottom) and active conversations (starters appear below the last message, giving the user the option to ask about the current product).

### D4: Service Compatibility

The implementation uses the exact `@weni/webchat-service` 1.9.0 API:
- `service.getStarters(productData)` — only called when `service.isConnected()` is true
- `service.clearStarters()` — called on navigation away from PDP
- `service.on('starters:received', handler)` — listen for results
- `service.on('starters:error', handler)` — handle errors silently
- `service.setContext(contextString)` — set SKU-level context as a formatted string

### D5: Two-Layer Data Extraction

1. **Product-level** (for starters generation): `account`, `linkText`, `productName`, `description`, `brand`, `attributes` → sent to `getStarters()`
2. **SKU-level** (for AI context): selected variant's name, price, availability, image, specifications → sent to `setContext()`

This separation ensures starters are cached per-product (efficient) while context is always variant-specific (accurate).

### D6: Reusing Existing Components

- **Button**: `src/components/common/Button.jsx` is reused for starter buttons, following the constitution's component reuse mandate.
- **CSS Variables**: Compact buttons use custom styling (dark background per screenshots). Full buttons use `--weni-main-color` for border, matching the primary color theming pattern from quick replies.
- **Animations**: Reuse existing `weni-slide-in` and `weni-slide-out` keyframes from `animations.scss`.
